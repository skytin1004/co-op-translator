## ការតំឡើងបរិបទអភិវឌ្ឍន៍

ដើម្បីតំឡើងបរិបទអភិវឌ្ឍន៍សម្រាប់គម្រោងនេះ យើងណែនាំឲ្យប្រើ Poetry សម្រាប់គ្រប់គ្រងខ្ទង់ការពឹង
យើងប្រើ `pyproject.toml` សម្រាប់គ្រប់គ្រងខ្ទង់ការពឹងគម្រោង ហើយដូច្នេះ ដើម្បីដំឡើងខ្ទង់ការពឹង អ្នកគួរប្រើ Poetry។

### ការសាកល្បងដោយដៃ

មុនពេលដាក់ស្នើ PR វាសំខាន់ក្នុងការសាកល្បងមុខងারបកប្រែក្នុងឯកសារពិតប្រាកដៈ

1. បង្កើតថតសាកល្បងមួយនៅក្នុងថតគោល ៖
    ```bash
    mkdir test_docs
    ```

2. ចម្លងឯកសារបកប្រែ markdown និងរូបភាពដែលអ្នកចង់បកប្រែទៅក្នុងថតសាកល្បង។ ឧទាហរណ៍ ៖
    ```bash
    cp /path/to/your/docs/*.md test_docs/
    cp /path/to/your/images/*.png test_docs/
    ```

3. ដំឡើងកញ្ចប់ក្នុងម៉ាស៊ីនក្នុងស្រុក៖
    ```bash
    pip install -e .
    ```

4. រត់ Co-op Translator លើឯកសារសាកល្បងរបស់អ្នក៖
    ```bash
    python -m co_op_translator --language-codes ko --root-dir test_docs
    ```

5. ពិនិត្យឯកសារបកប្រែដែលនៅក្នុង `test_docs/translations` និង `test_docs/translated_images` ដើម្បីផ្ទៀងផ្ទាត់៖
   - គុណភាពនៃការបកប្រែ
   - ការអត្ថាធិប្បាយ metadata ត្រឹមត្រូវ
   - រចនាសម្ព័ន្ធ markdown ដើមត្រូវបានរក្សាទុក
   - តំណ និងរូបភាពដំណើរការបានត្រឹមត្រូវ

ការសាកល្បងដោយដៃនេះជួយធានាថាការផ្លាស់ប្តូររបស់អ្នកដំណើរការល្អនៅក្នុងស្ថានការណ៍ពិត។

## រត់ Co-op Translator

ដើម្បីរត់ Co-op Translator ដោយប្រើ Poetry ក្នុងបរិបទរបស់អ្នក សូមអនុវត្តជំហានខាងក្រោម៖

1. ទៅកាន់ថតដែលអ្នកចង់ធ្វើការបកប្រែសាកល្បង ឬបង្កើតថតប្រាក់ពន្ធណាមួយសម្រាប់ការសាកល្បង។

2. ប្រតិបត្តិការបញ្ជារខាងក្រោម។ នៅទីនេះ `-l ko` ជាកូដភាសាដែលអ្នកចង់បកប្រែ។ ប៊ូតុង `-d` បង្ហាញថាគឺជា របៀបកែតម្រូវ។

    ```bash
    poetry run co-op-translator translate -l ko -d
    ```

> [!NOTE]
> សូមប្រាកដថាបរិបទ Poetry របស់អ្នកបានបើក (poetry shell) មុនពេលរត់បញ្ជា។

## ចូលរួមបញ្ចូលភាសាថ្មី

យើងស្វាគមន៍ការចូលរួមបន្ថែមការគាំទ្រសម្រាប់ភាសាថ្មីៗ។ មុនពេលបើក PR សូមអនុវត្តជំហានខាងក្រោមដើម្បីធ្វើឲ្យការត្រួតពិនិត្យរលូន។

1. เพิ่มភាសាក្នុង mapping អក្សរ
   - កែសម្រួល `src/co_op_translator/fonts/font_language_mappings.yml`
   - បន្ថែមអាំងត្រីត្យុងមួយមាន:
     - `code`: កូដភាសាជា ISO (ឧ. `vi`)
     - `name`: ឈ្មោះបង្ហាញស្រួលយល់សម្រាប់មនុស្ស
     - `font`: ភាពអក្សរមួយដែលមាននៅក្នុង `src/co_op_translator/fonts/` ដែលគាំទ្របំណង់អក្សរ
     - `rtl`: `true` ប្រសិនបើភាសានេះពីស្តាំទៅឆ្វេង ហើយ ប្រសិនបើមិនមែន `false`

2. រាប់បញ្ចូលឯកសាររូបអក្សរដែលត្រូវការ (បើចាំបាច់)
   - ប្រសិនបើត្រូវការ font ថ្មី បញ្ចាក់អាជ្ញាប័ណ្ណដែលត្រូវអាចចែកចាយដោយសហគមន៍
   - បន្ថែមឯកសារ font ទៅក្នុង `src/co_op_translator/fonts/`

3. ពិនិត្យក្នុងស្រុក
   - បញ្ចប់ការបកប្រែសម្រាប់គំរូតូចៗ (Markdown, រូបភាព និងកំណត់ត្រាដូចតម្រូវ)
   - ពិនិត្យលទ្ធផលថាបង្ហាញបានត្រឹមត្រូវ រួមមាន font និងរចនាសម្ព័ន្ធ rtl បើមាន

4. ធ្វើបច្ចុប្បន្នភាពឯកសារ
   - ប្រាកដថាភាសាត្រូវបង្ហាញនៅក្នុង `getting_started/supported-languages.md`
   - គ្មានការផ្លាស់ប្តូរចាំបាច់ទៅ `getting_started/README_languages_template.md` ព្រោះវាត្រូវបានបង្កើតពីបញ្ជីគាំទ្រ

5. បើក PR
   - ពិពណ៌នាអំពីភាសាថ្មី និងការពិចារណាអាជ្ញាប័ណ្ណ/អក្សរ
   - ជាប់ផ្ទាប់ជាមួយរូបភាពនៃលទ្ធផលបកប្រែ ប្រសិនបើអាចធ្វើបាន

ឧទាហរណ៍ YAML entry:

```yaml
new_lang(code):
  name: "New Language"
  font: "NotoSans-Medium.ttf"
  rtl: false
```

### សាកល្បងភាសាថ្មី

អ្នកអាចសាកល្បងភាសាថ្មីដោយការរត់បញ្ជាច្រើនដូចខាងក្រោម៖

```bash
# បង្កើត និង បើកបរបរិយាកាសវីរុត
python -m venv .venv
# វីនដូរ
.venv\Scripts\activate
# macOS/Linux
source .venv/bin/activate
# ដំឡើងកញ្ចប់អភិវឌ្ឍន៍
pip install -e .
# ប្រតិបត្តិការប្រែសម្រួល
translate -l "new_lang"
```

## អ្នកថែទាំ

### សារ commit និងយុទ្ធសាស្ត្រការរួមបញ្ចូល

ដើម្បីធានាថាប្រវត្តិ commit របស់គម្រោងយើងមានរបៀបស្អាត និងច្បាស់លាស់ យើងអនុវត្តទ្រង់ទ្រាយសារប្រាប់ commit ជាក់លាក់ **សម្រាប់ commit ចុងក្រោយ** នៅពេលប្រើយុទ្ធសាស្ត្រ **Squash and Merge**។

ពេល pull request (PR) ត្រូវបានបញ្ចូល commit ជាក់លាក់នឹងត្រូវបានសំលាប់ជាការបញ្ចូល commit តែមួយ។ សារប្រាប់ commit ចុងក្រោយគួរត្រូវតាមទ្រង់ទ្រាយខាងក្រោម ដើម្បីរក្សាប្រវត្តិស្អាត និងស្រួលយល់។

#### ទ្រង់ទ្រាយសារប្រាប់ commit (សម្រាប់ squash and merge)

យើងប្រើទ្រង់ទ្រាយខាងក្រោមសម្រាប់សារប្រាប់ commit៖

```bash
<type>: <description> (#<លេខ PR>)
```

- **type**: បញ្ជាក់ប្រភេទ commit ។ យើងប្រើ type ខាងក្រោម៖
  - `Docs`: សម្រាប់ការអាប់ដេតឯកសារ។
  - `Build`: សម្រាប់ផ្លាស់ប្តូរពាក់ព័ន្ធប្រព័ន្ធសាងសង់ ឬខ្ទង់ការពឹង រួមទាំងការអាប់ដេតឯកសារកំណត់កំណត់ ការបង់ការងារ CI ឬ Dockerfile។
  - `Core`: សម្រាប់ការផ្លាស់ប្តូរមុខងារស្នូល គឺផ្នែក core គម្រោង ជាពិសេសឯកសារក្នុងថត `src/co_op_translator/core`។

- **description**: សេចក្ដីសង្ខេបខ្លីអំពីការផ្លាស់ប្តូរ។
- **PR number**: លេខ pull request ដែលទាក់ទងជាមួយ commit។

**ឧទាហរណ៍**៖

- `Docs: Update installation instructions for clarity (#50)`
- `Core: Improve handling of image translation (#60)`

> [!NOTE]
> បច្ចុប្បន្ននេះ **`Docs`**, **`Core`**, និង **`Build`** ពាក្យដាក់ជាមុនត្រូវបានបន្ថែមដោយស្វ័យប្រវត្តិទៅចំណងជើង PR ដោយផ្អែកលើប្លាក​ឯលប្រភេទកូដដែលបានបំលែង។ ត្រឹមតែពិនិត្យឲ្យប្រាកដថាប្លាកត្រឹមត្រូវ អ្នកមិនចាំបាច់កែសម្រួលចំណងជើង PR ដោយដៃទេ។

#### គោលការណ៍រួមបញ្ចូល

យើងប្រើ **Squash and Merge** ជាយុទ្ធសាស្ត្រ​លំនាំដើមសម្រាប់ pull request។ វានឹងធានាថាសារប្រាប់ commit ត្រូវតាមទ្រង់ទ្រាយរបស់យើង ទោះបី commit ជាក់លាក់មិនត្រូវទ្រង់ទាយក៏ដោយ។

**ហេតុផល**:

- ប្រវត្តិសាស្ត្រគម្រោងស្អាត និងត្រង់
- ទ្រង់ទាយសារប្រាប់ commit ទ្ងោក
- កាត់បន្ថយសំឡេងច្រេីនពី commit តូចៗ (ឧ. "fix typo")

ពេលបញ្ចូល commit សូមធានាថាសារប្រាប់ commit ចុងក្រោយត្រូវតាមទ្រង់ទាយខាងលើ។

**ឧទាហរណ៍ Squash and Merge**  
បើ PR មាន commit ដូចខាងក្រោម៖

- `fix typo`
- `update README`
- `adjust formatting`

ពួកវាគួរត្រូវបានសំលាប់ជា៖  
`Docs: Improve documentation clarity and formatting (#65)`

### ដំណើរការចេញផ្សាយ

ផ្នែកនេះពិពណ៌នារបៀបសាមញ្ញបំផុតសម្រាប់អ្នកថែទាំក្នុងការចេញផ្សាយកំណែថ្មីនៃ Co-op Translator។

#### 1. បង្កើន version ក្នុង `pyproject.toml`

1. កំណត់លេខកំណែបន្ទាប់ (យើងអនុវត្ត semantic versioning: `MAJOR.MINOR.PATCH`)។
2. កែសម្រួល `pyproject.toml` និងបន្ទាន់សម័យវាល `version` នៅក្រោម `[tool.poetry]`។
3. បើក pull request ផ្សេងមួយសម្រាប់បម្លែងលេខកំណែ (និងឯកសារ lock/metadata ដែលមានកំណែប្រែដោយស្វ័យប្រវត្តិ)។
4. បន្ទាប់ពីត្រួតពិនិត្យ អនុវត្តយុទ្ធសាស្រ្ត **Squash and Merge** និងប្រាកដថាសារប្រាប់ commit ចុងក្រោយ​ត្រូវតាមទ្រង់ទាយខាងលើ។

#### 2. បង្កើត GitHub Release

1. ចូលទំព័រ repository GitHub ហើយបើក **Releases** → **Draft a new release**។
2. បង្កើត tag ថ្មី (ឧ. `v0.13.0`) ពីbranch `main`។
3. កំណត់ចំណងជើង release ទៅបញ្ជាក់ version ដូចគ្នា (ឧ. `v0.13.0`)។
4. ចុច **Generate release notes** ដើម្បីបញ្ចូល changelog ដោយស្វ័យប្រវត្តិ។
5. ជាជម្រើសកែសម្រួលអត្ថបទ (ឧ. ដើម្បីបង្ហាញភាសាថ្មីដែលគាំទ្រ ឬការផ្លាស់ប្តូរមានសារៈសំខាន់)។
6. ដាក់ផ្សាយ release។
